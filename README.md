# نظام إدارة شركة الحراسات الأمنية (Security ERP)

مشروع Laravel لإدارة شركة حراسات: الموظفين، الرواتب، العقود، الفواتير، المبيعات، السندات، والتكاليف.

## ✅ الحالة الحالية: Phase 1 + Phase 2

### Phase 1 (البنية الأساسية)
- جدول **users** بأدوار: `admin, hr, accountant, manager, supervisor`
- جدول **employees**: الموظفين والحراس (بيانات، هوية، تواريخ، راتب أساسي، تنبيه انتهاء الهوية/الإقامة)
- جدول **client_companies**: الشركات العميلة المتعاقد معها
- جدول **sites**: المواقع/الفروع التابعة لكل شركة عميلة
- جدول **employee_site_assignments**: تعيين الحراس على المواقع بالشفت (صباحي/مسائي/ليلي/يوم كامل)
- Middleware للتحقق من الصلاحيات حسب الدور (`role:admin,hr...`)
- Controllers + Routes كاملة (CRUD) للموظفين، الشركات، المواقع، والتعيينات
- Seeder ببيانات تجريبية + مستخدم أدمن أولي

### Phase 2 (العقود)
- جدول **contracts**: عقود الحراسة بـ 3 أنواع (`monthly` شهري / `annual` سنوي / `lump_sum` مقطوع)
  ودورة فوترة مستقلة (`monthly/quarterly/annual/one_time`) — أي يمكن عقد سنوي بفوترة شهرية مثلًا
- جدول **contract_sites**: ربط العقد بموقع أو أكثر، مع عدد الحراس وسعر الحارس لكل موقع
  (السعر الإجمالي `site_total` يُحسب تلقائيًا عند الحفظ)
- منطق تلقائي في Model العقد:
  - `isExpiringSoon()` / `isExpired()` لحساب حالة العقد
  - `calculateNextInvoiceDate()` لحساب موعد الفاتورة القادمة حسب دورة الفوترة
  - `isDueForInvoicing()` لتحديد العقود المستحقة للفوترة الآن (سيُستخدم في Phase 3 لتوليد الفواتير تلقائيًا)
- أمر Artisan مجدول يوميًا `contracts:refresh-status`:
  - يفعّل العقود التي حان تاريخ بدايتها
  - يُنهي العقود المنتهية تلقائيًا
  - يطبع تنبيهًا بالعقود القريبة من الانتهاء (خلال 30 يوم) — جاهز لاحقًا للربط بإشعار بريد/واتساب
- Controller + Routes كاملة لإنشاء/تعديل/إلغاء/تفعيل العقود مع مواقعها وأسعارها دفعة واحدة

### Phase 3 (الفواتير والمبيعات)
- جدول **suppliers**: الموردين (لفواتير الشراء: زي، معدات، صيانة...)
- جدول **sales_invoices** + **sales_invoice_items**: فواتير البيع الصادرة للشركات العميلة
  - `source`: `manual` (فاتورة يدوية لخدمة إضافية) أو `auto_contract` (مولّدة تلقائيًا من عقد)
  - إعادة حساب `subtotal / tax_amount / total_amount` تلقائيًا عند أي تعديل على البنود (Model Events)
  - `status` (unpaid/partially_paid/paid/cancelled) تُحدَّث تلقائيًا حسب `paid_amount` (سيتم تغذيته من سندات القبض في Phase 4)
- جدول **purchase_invoices** + **purchase_invoice_items**: فواتير الشراء من الموردين، بنفس منطق الحساب التلقائي والتصنيف (زي/معدات/صيانة/مرافق/أخرى)
- **خدمة `ContractInvoiceGenerator`**: تفحص كل العقود المستحقة للفوترة (`Contract::dueForInvoicing()`) وتُنشئ فاتورة بيع تلقائيًا ببند لكل موقع في العقد، ثم تُحدّث موعد الفوترة القادم للعقد
- أمر Artisan مجدول يوميًا `invoices:generate-from-contracts` (الساعة 6:15 صباحًا، بعد تحديث حالات العقود مباشرة) — **هذا هو قلب "نظام المبيعات" الآلي**: كل عقد شهري نشط يُصدر فاتورته تلقائيًا دون تدخل بشري
- Controllers + Routes لفواتير البيع (يدوي)، فواتير الشراء، والموردين — بصلاحية أدمن/محاسب فقط
- تحديث السيدر بمورد وفاتورة شراء تجريبية + توليد فاتورة بيع تلقائية فعلية من العقد التجريبي

### Phase 4 (الصناديق والسندات)
- جدول **cash_accounts**: صناديق نقدية وحسابات بنكية متعددة، كل حساب له `current_balance` يُحدَّث تلقائيًا
- جدول **receipt_vouchers** (سندات القبض): تحصيل مبالغ من العملاء، تُربط اختياريًا بفاتورة بيع محددة
- جدول **payment_vouchers** (سندات الصرف): صرف مبالغ لموردين (مرتبط بفاتورة شراء)، رواتب (سيُربط بالموظف في Phase 5)، أو **مصاريف/تكاليف تشغيلية عامة** (`purpose = expense`) مصنّفة بحقل `cost_category` (إيجار، وقود، مرافق، صيانة، نقل، رسوم حكومية، أخرى) — **هذا هو مسار تسجيل "التكاليف" الذي طلبته**
- كل سند عند تأكيده (`applyEffects()`):
  - يُحدّث رصيد الصندوق/البنك تلقائيًا (زيادة عند القبض، نقصان عند الصرف)
  - يُحدّث `paid_amount` وحالة السداد بالفاتورة المرتبطة تلقائيًا (عبر `refreshPaymentStatus()` من Phase 3)
- **حماية من الأخطاء**: لا يمكن تحصيل مبلغ أكبر من المتبقي على الفاتورة، ولا صرف مبلغ أكبر من رصيد الصندوق المتاح
- **الإلغاء بدل الحذف**: أي سند يمكن إلغاؤه (`cancel()`) وهذا يعكس تلقائيًا أثره بالكامل على الصندوق والفاتورة، مع الاحتفاظ بالسجل لأغراض التدقيق
- Controllers + Routes للصناديق والسندين — بصلاحية أدمن/محاسب فقط
- تحديث السيدر بصندوق نقدي، سند قبض جزئي (تحصيل 50% من فاتورة تلقائية)، وسند صرف كامل (سداد فاتورة الزي الموحد)

### Phase 5 (الرواتب والسلف)
- جدول **employee_advances** (السلف): مبلغ السلفة، قسط الخصم الشهري، والرصيد المتبقي (تُحدَّث تلقائيًا مع كل تشغيل رواتب)
- جدول **payroll_runs** (تشغيل رواتب شهري): حالة `draft → approved → paid`، تشغيل واحد فقط لكل شهر/سنة
- جدول **payroll_items** (بند راتب كل موظف): `net_salary = الأساسي + البدلات − قسط السلفة − خصومات أخرى` (يُحسب تلقائيًا)
- **خدمة `PayrollService`** تدير دورة العمل الكاملة:
  1. `createRun()` — يولّد بندًا تلقائيًا لكل موظف نشط (بالراتب الأساسي وخصم قسط أي سلفة نشطة)
  2. تعديل يدوي اختياري على البدلات/الخصومات لكل موظف قبل الاعتماد (التشغيل بحالة `draft` فقط)
  3. `approve()` — اعتماد التشغيل ليصبح جاهزًا للصرف
  4. `payItem()` / `payAll()` — **يُنشئ سند صرف تلقائيًا** لكل موظف (`purpose = salary`) من الصندوق المختار، يخصم رصيد الصندوق، ويخصم قسط السلفة الفعلي من رصيدها (وتُغلق تلقائيًا عند اكتمال السداد)
- حماية: لا يمكن التعديل بعد الاعتماد، ولا الصرف من رصيد صندوق غير كافٍ، ولا صرف نفس البند مرتين
- أمر مجدول شهريًا `payroll:generate-monthly` (أول كل شهر) يُنشئ مسودة تشغيل رواتب الشهر الجديد تلقائيًا لمراجعتها واعتمادها
- تحديث السيدر بموظف ثانٍ وسلفة نشطة له، وتشغيل رواتب كامل (إنشاء → اعتماد → صرف الكل) كمثال تطبيقي شامل

### Phase 6 (التقارير والتصدير) — المرحلة الأخيرة ✅
- **`ReportService`**: خدمة مركزية تجمع بيانات كل التقارير:
  - `financialSummary()`: ملخص مالي لفترة (مبيعات مفوترة/محصَّلة، مشتريات، مدفوعات موردين، رواتب، مصاريف مصنّفة، صافي التدفق النقدي، أرصدة الصناديق)
  - `clientStatement()` / `supplierStatement()`: كشف حساب بفواتير وسندات مرتبة زمنيًا مع **رصيد متحرك (Running Balance)**
  - `contractsPerformance()`: الإيراد الشهري المتوقع من كل العقود النشطة + قائمة العقود القريبة من الانتهاء
  - `payrollSummary()`: ملخص تشغيل رواتب شهر معين (إجمالي الأساسي/البدلات/الخصومات/الصافي وعدد المدفوع)
- **لوحة تحكم رئيسية** (`/dashboard`) بأهم المؤشرات اللحظية (موظفين نشطين، عقود نشطة وقريبة الانتهاء، فواتير غير محصّلة ومتأخرة)
- **تصدير PDF** (عبر `barryvdh/laravel-dompdf`): الملخص المالي وكشف حساب العميل
- **تصدير Excel** (عبر `maatwebsite/excel`): الملخص المالي، كشف حساب العميل، وكشف حساب المورد
- تمت إضافة **Views** فعلية لأول مرة في هذه المرحلة (تخطيط عام + صفحات التقارير) بواجهة عربية RTL بسيطة تعمل مباشرة، لتكون المشروع قابلًا للتجربة البصرية فورًا وليس فقط كبنية خلفية

### Phase 7 (مستفاد من مراجعة نسخة ثانية) ✅
تحليل مشروع آخر أُعدَّ لنفس الغرض (بواجهة جميلة لكن بدون تنفيذ خلفي فعلي حقيقي) كشف عدة أفكار تصميم جيدة في مخطط بياناته. تم تبنّي التالي فعليًا داخل نظامنا العامل:

- **الفروع (`branches`)**: تعدد فروع الشركة نفسها بأكثر من مدينة، مربوط بـ`branch_id` في: المستخدمين، الموظفين، الشركات العميلة، العقود، الصناديق، الموردين
- **بيانات موظف أدق**: فصل رقم الجواز عن الإقامة بتاريخي انتهاء مستقلين، رقم التأمينات الاجتماعية (GOSI)، رقم وتاريخ انتهاء رخصة الحراسة الأمنية، بيانات بنكية (بنك/حساب/آيبان) + جدول **`employee_documents`** لوثائق متعددة لكل موظف بدل مرفق واحد
- **الحضور والانصراف الحقيقي (`attendances`)**: أهم فجوة كانت موثّقة سابقًا كـ"غير مدعومة" — تسجيل يومي فعلي (حضور/غياب/تأخير/إجازة) بالشفت والساعات الإضافية، مع Controller كامل (تسجيل، فلترة، ملخص شهري)
- **ربط الحضور بالرواتب فعليًا (تحسين لا يملكه أي من المشروعين)**: `PayrollService::createRun()` أصبح يقرأ سجلات حضور الموظف الفعلية لذلك الشهر ويحسب تلقائيًا: خصم أيام الغياب (بالمعدل اليومي) ومبلغ العمل الإضافي (بمعدل 1.5×) — بدل إدخال هذه الأرقام يدويًا فقط
- **بدلات راتب مفصّلة**: سكن/نقل/طعام كحقول مستقلة على الموظف (قيمة افتراضية) وعلى كل بند راتب شهري (القيمة الفعلية المدفوعة)
- **مولّد أرقام تسلسلي آمن (`NumberSequence`)**: يستبدل منطق `count()+1` القديم الذي كان عرضة للتعارض عند حذف سجل؛ الآن عداد ذري محمي بـ`lockForUpdate` لكل نوع مستند (فاتورة بيع، سند قبض، سند صرف، طلب دفع) مع تصفير تلقائي كل سنة
- **سجل تدقيق عام (`AuditLog` + Trait `Auditable`)**: من عدّل ماذا ومتى، مفعَّل تلقائيًا بسطر واحد (`use Auditable;`) على 7 نماذج حساسة: العقود، فواتير البيع والشراء، سندات القبض والصرف، تشغيلات الرواتب، السلف
- **جاهزية ZATCA**: حقول فارغة على فاتورة البيع (`zatca_status`, `zatca_qr_code`, `zatca_uuid`) بانتظار ربط فعلي مستقبلي — جاهزية فقط بدون منطق وهمي مضلِّل
- **حد ائتماني للعميل — مُفعَّل فعليًا وليس شكليًا**: عكس المشروع المرجعي الذي يخزّن `creditLimit` دون استخدامه، هنا `ClientCompany::hasAvailableCredit()` يُستدعى فعليًا ويمنع إصدار فاتورة يدوية تتجاوز حد العميل
- **أمر الدفع (`payment_orders`)**: مرحلة "طلب واعتماد" منفصلة قبل إنشاء سند الصرف الفعلي — لا يمكن ربط سند صرف بأمر دفع غير معتمد، ويُحدَّث الأمر تلقائيًا لحالة "مدفوع" عند إنشاء السند

**ما لم يُنقَل عمدًا من المشروع الآخر** (لأنها نقاط ضعف فيه وليست مزايا): الجدول الوهمي في الذاكرة بدل قاعدة بيانات حقيقية، مسارات API لا تعمل (Next.js غير مثبّت أصلًا)، ميزة تحليل "حوادث أمنية" بالذكاء الاصطناعي غير المتعلقة بالمشروع.

### Phase 8 (تكامل الموقع الجغرافي GPS) ✅
- حقول إحداثيات + نطاق مسموح (`geofence_radius_meters`، افتراضيًا 150م) على كل **موقع حراسة** (`sites`)
- حقول إحداثيات فعلية عند لحظة الحضور (`check_in_lat/lng`) + المسافة المحسوبة (`check_in_distance_meters`) + نتيجة التحقق (`is_within_geofence`) على كل **سجل حضور**
- **`GeoService::distanceMeters()`**: حساب المسافة بمعادلة Haversine بين نقطتين، دون أي مكتبة خارجية
- عند تسجيل الحضور، إن تتوفر إحداثيات الموقع المسجَّل + إحداثيات الحضور الفعلية، يُحسَب التحقق **تلقائيًا** ويُخزَّن (لا حاجة لأي إدخال يدوي إضافي)
- **صفحة تسجيل حضور من الجوال** (`attendance/create`) تستخدم `navigator.geolocation` (متصفح الجوال مباشرة) لالتقاط الموقع الفعلي قبل تفعيل زر الإرسال — **لا تحتاج تطبيقًا منفصلًا ولا جهازًا خاصًا**، بخلاف تكامل البصمة الذي يعتمد على ماركة جهاز محددة
- تنبيه واضح في الرسالة وبجدول السجلات عند كون الحارس **خارج نطاق موقعه** المسموح وقت تسجيل الحضور

## ⏭️ ما تبقى (خارج نطاق هذه الجولة)
- ربط الأدوار/الصلاحيات بالفرع (حاليًا `branch_id` مضاف للبيانات فقط، دون فلترة تلقائية للاستعلامات حسب فرع المستخدم — يحتاج Middleware/Global Scope إضافي)
- واجهات Blade لبعض الشاشات الأخرى (أوامر الدفع، وثائق الموظف) — لم تُبنَ Views بعد لهذه الوحدات تحديدًا
- تكامل ZATCA فعلي (الحقول جاهزة فقط)
- تكامل البصمة الفعلي (يحتاج تحديد ماركة الجهاز أولًا لاختيار البروتوكول المناسب — ADMS Push أو SDK محلي)

## 🚀 خطوات التركيب (على جهازك/سيرفرك)

المتطلبات: PHP >= 8.2، Composer، MySQL >= 8، Node.js (لبناء الواجهات لاحقًا)

```bash
# 1) أنشئ مشروع Laravel جديد فارغ في نفس المجلد (لتوليد الملفات الأساسية المفقودة هنا مثل .env, artisan, vendor)
composer create-project laravel/laravel security-erp-base
# انسخ محتويات هذا المجلد (app, database, routes) فوق مشروعك الجديد

# 2) تثبيت الحزم
cd security-erp-base
composer install

# 3) تثبيت نظام تسجيل الدخول الجاهز (Laravel Breeze)
composer require laravel/breeze --dev
php artisan breeze:install blade
npm install && npm run build

# 4) إعداد قاعدة البيانات في ملف .env
DB_DATABASE=security_erp
DB_USERNAME=root
DB_PASSWORD=

# 5) تشغيل المايجريشن والبيانات التجريبية
php artisan migrate
php artisan db:seed --class=Phase1Seeder

# 6) تسجيل الـ Middleware في bootstrap/app.php (Laravel 11):
# $middleware->alias(['role' => \App\Http\Middleware\CheckRole::class]);

# 7) تشغيل السيرفر محليًا للتجربة
php artisan serve

# 8) (اختياري الآن) تفعيل الجدولة اليومية لتحديث حالات العقود
php artisan schedule:work

# 9) نشر إعدادات حزم PDF و Excel (بعد composer install، تُثبت تلقائيًا مع الحزم في composer.json)
php artisan vendor:publish --provider="Barryvdh\DomPDF\ServiceProvider"
php artisan vendor:publish --provider="Maatwebsite\Excel\ExcelServiceProvider"
```

بيانات الدخول التجريبية بعد الـ Seeder:
- البريد: `admin@security-erp.local`
- كلمة المرور: `password` (غيّرها فورًا)

## 📌 ملاحظة مهمة
لم يتم إنشاء ملفات Views (Blade) بعد في هذه المرحلة — الأولوية كانت للبنية الخلفية (Models/Migrations/Controllers/Routes). في Phase 2 سنبدأ بإضافة الواجهات (أو يمكن ربطها بلوحة تحكم جاهزة مثل Filament لتسريع الواجهات إن رغبت).
